============================================================
OPERATION 1 — RECONCILE
============================================================

Goal: find contradictions in the vault and resolve by marking older notes
as superseded. NEVER delete notes; preserve everything as audit trail.

Process:

1. For each canonical folder (Decisions/, Projects/, Ideas/, Research/),
   scan note frontmatter and `## For future Claude` preambles for the
   same `[[wikilinked-entity]]` or same topic appearing in multiple notes.

2. Identify contradictions: notes that state opposing facts, reversed
   commitments, or supersede earlier positions about the same entity.

3. For each contradiction, decide which note is current (use `date:`
   frontmatter; newer wins by default).

4. Edit the older note's frontmatter:
       status: superseded
       superseded_by: <relative-path-to-newer-note>
   And append to its body (if not already present):
       ## Updates
       - <today's-date> — Superseded by [[<newer-note-stem>]].
       <one-sentence reason>

5. DO NOT modify the newer note. DO NOT delete the older note.

If you find zero contradictions, that's a fine result — print
`reconcile: 0 contradictions` and continue.

============================================================
OPERATION 2 — CONSOLIDATE (rename + merge + cross-link)
============================================================

Goal: keep the vault from accumulating timestamped, duplicate, and
orphaned notes. Three sub-operations, applied to notes shown above.

DO NOT write to the legacy {{DIARY_ROOT}}/Synthesis/ folder. Standalone
synthesis pages are not produced anymore — consolidation happens inside
the canonical folders themselves.

------------------------------------------------------------
2a. RENAME timestamped filenames (unconditional)
------------------------------------------------------------

For any note in a canonical folder whose filename matches a timestamp
pattern (YYYY-MM-DDTHH-MM-SS.md or similar), generate a kebab-case slug
from the note's content (3–5 words that capture the topic). Rename the
file via `mv` to <slug>.md.

- Preserve the original ID by adding `original_id: <timestamp>` to the
  frontmatter.
- If a file with that exact slug already exists in the same folder AND
  is about the same topic, skip rename and go to 2b (MERGE).
- If a file with that exact slug exists but is about a different topic,
  pick a disambiguating slug (add a distinguishing word).
- Don't rename files that already have descriptive slugs.

------------------------------------------------------------
2b. MERGE within a folder (≥80% confidence)
------------------------------------------------------------

Identify clusters of ≥2 notes in the SAME canonical folder that are
clearly about the same topic. Threshold: ≥80% confident — uncertain
clusters stay separate.

For each cluster:

1. Pick the canonical: the note with the most substantive content (oldest
   if tied).
2. Apply 2a to rename the canonical to a topic-slug if it's still
   timestamped.
3. MERGE bodies: append each non-canonical source's verbatim content to
   the canonical under a new `## Original notes` section, each subsection
   prefixed by `### From <slug-or-id> (<date>)`.
4. Revise the canonical's `## For future Claude` preamble to reflect ALL
   inputs (the merged perspective, not just the canonical's original
   take). Keep it 2–3 lines.
5. For each non-canonical source note, REWRITE the body to a stub
   redirect — preserve its frontmatter and add `merged_into:`:

       ---
       type: <original type>
       date: <original date>
       original_id: <if present>
       merged_into: <relative-path-to-canonical>
       ai-first: true
       ---

       Moved → [[<canonical-slug>]].

   Existing `[[wikilinks]]` pointing at this path still resolve (Obsidian
   shows the stub), and the stub points readers at the canonical.

------------------------------------------------------------
2c. CROSS-FOLDER WIKILINK (don't merge — just connect)
------------------------------------------------------------

When a note in one folder is clearly about the same topic as a note in a
DIFFERENT canonical folder (e.g. Projects/<x> and Ideas/<x>), DON'T
merge them. Add bidirectional wikilinks via short labeled blockquotes,
placed right after the `## For future Claude` preamble.

Recommended labels:

- A later-stage note pointing back to its origin idea/note:
      > Origin: [[Ideas/<x>]]
- An earlier-stage note pointing forward to what it became:
      > Became: [[Projects/<x>]]   (or  > Decided: [[Decisions/<x>]] )
- Same-tier or unclear direction:

DO NOT trigger the `graduate` skill behavior here. Maintain only adds
wikilinks and a one-line label — it never creates new project scaffolding,
never copies content between folders, never deletes the originals. The
two notes stay independent, just visibly linked.
