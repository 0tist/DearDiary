============================================================
OPERATION 1 — RECONCILE
============================================================

Goal: find contradictions in the vault and resolve by marking older notes
as superseded. NEVER delete notes; preserve everything as audit trail.

Process:

1. For each canonical folder (Decisions/, Projects/, Ideas/, Research/),
   scan note frontmatter and `## For future Claude` preambles for the
   same `[[wikilinked-entity]]` or same topic appearing in multiple notes.

2. Identify contradictions: notes that state opposing facts, reversed
   commitments, or supersede earlier positions about the same entity.

3. For each contradiction, decide which note is current (use `date:`
   frontmatter; newer wins by default).

4. Edit the older note's frontmatter:
       status: superseded
       superseded_by: <relative-path-to-newer-note>
   And append to its body (if not already present):
       ## Updates
       - <today's-date> — Superseded by [[<newer-note-stem>]].
       <one-sentence reason>

5. DO NOT modify the newer note. DO NOT delete the older note.

If you find zero contradictions, that's a fine result — print
`reconcile: 0 contradictions` and continue.

============================================================
OPERATION 2 — CONSOLIDATE (rename + merge + cross-link)
============================================================

Goal: keep the vault from accumulating timestamped, duplicate, and
orphaned notes. Three sub-operations, applied to notes shown above.

DO NOT touch the legacy {{DIARY_ROOT}}/Synthesis/ folder. Don't create
new files there; don't edit existing files there. Standalone synthesis
pages are deprecated — consolidation happens inside the canonical
folders themselves.

HARD RULE on wikilinks: every `[[link]]` you emit must resolve to a
file that actually exists in the vault (use the renamed slug, not the
old timestamped stem, and never invent a target that isn't there). If
the link would 404, either rename the file to that slug first (2a),
write the target as part of a merge (2b), or skip the link entirely.

------------------------------------------------------------
2a. RENAME timestamped filenames (unconditional)
------------------------------------------------------------

For any note in a canonical folder whose filename matches a timestamp
pattern (YYYY-MM-DDTHH-MM-SS.md or similar), generate a kebab-case slug
from the note's content (3–5 words that capture the topic). Rename the
file via `mv` to <slug>.md.

- Preserve the original ID by adding `original_id: <timestamp>` to the
  frontmatter.
- If a file with that exact slug already exists in the same folder AND
  is about the same topic, skip rename and go to 2b (MERGE).
- If a file with that exact slug exists but is about a different topic,
  pick a disambiguating slug (add a distinguishing word).
- Don't rename files that already have descriptive slugs.

------------------------------------------------------------
2b. MERGE within a folder (high threshold — same idea, near-duplicate)
------------------------------------------------------------

ONLY merge when ≥2 notes in the SAME canonical folder are essentially
the SAME idea — the user wrote the same thought twice, or two notes
cover overlapping ground with no distinct angle. Threshold: high. If
each note has its own substantive angle, paper, hypothesis, or
estimate, that's NOT a merge candidate — those are RELATED notes
(see 2c below).

Examples of merge candidates:
  ✓ Two Ideas notes both about "microgreens automation at home" with
    overlapping content — merge.
  ✓ Three Daily/ entries from the same day captured by accident — merge.

Examples that are NOT merge candidates (use 2c instead):
  ✗ Three Ideas notes that each propose a DIFFERENT research approach
    (TTA, cross-modality, optimal control) to the same underlying
    goal — these are related angles, not duplicates. KEEP them separate
    and cross-link via 2c.

For each cluster:

1. Pick the canonical: the note with the most substantive content (oldest
   if tied).
2. Apply 2a to rename the canonical to a topic-slug if it's still
   timestamped.
3. MERGE bodies: append each non-canonical source's verbatim content to
   the canonical under a new `## Original notes` section, each subsection
   prefixed by `### From <slug-or-id> (<date>)`.
4. Revise the canonical's `## For future Claude` preamble to reflect ALL
   inputs (the merged perspective, not just the canonical's original
   take). Keep it 2–3 lines.
5. For each non-canonical source note, REWRITE the body to a stub
   redirect — preserve its frontmatter and add `merged_into:`:

       ---
       type: <original type>
       date: <original date>
       original_id: <if present>
       merged_into: <relative-path-to-canonical>
       ai-first: true
       ---

       Moved → [[<canonical-slug>]].

   Existing `[[wikilinks]]` pointing at this path still resolve (Obsidian
   shows the stub), and the stub points readers at the canonical.

------------------------------------------------------------
2c. WIKILINK related notes (within OR across folders — don't merge)
------------------------------------------------------------

When ≥2 notes are clearly RELATED (touch the same topic area, same
project, same entity, or are different angles on one underlying
problem) but are NOT merge candidates per 2b, add bidirectional
wikilinks. This applies BOTH within a single folder AND across folders.

Place the wikilink as a short labeled blockquote immediately after the
`## For future Claude` preamble of each related note (mutually — every
member of the cluster links to every other member).

Recommended labels (pick what fits the relationship):

- Cross-folder, later-stage note pointing back to its origin:
      > Origin: [[<earlier-slug>]]
- Cross-folder, earlier-stage note pointing forward:
      > Became: [[<later-slug>]]    or  > Decided: [[<decision-slug>]]
- Same-folder or unclear-direction, related angles:
      > See also: [[<other-slug>]]
- Multiple related siblings:
      > Related: [[<a>]], [[<b>]]

DO NOT trigger the `graduate` skill behavior here. Maintain only adds
wikilinks and a one-line label — it never creates new project
scaffolding, never copies content between folders, never deletes
notes. The notes stay independent files, just visibly linked.

Wikilink targets MUST resolve (see HARD RULE above). If you've renamed
a file as part of 2a, every link must use the new slug, not the old
timestamped stem.
