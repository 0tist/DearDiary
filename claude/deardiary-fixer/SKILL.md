# DearDiary Fixer

The user runs this repo as their personal Claude Code glue (hooks, scripts,
Tauri GUI, skills). They sometimes log ideas in the diary that are actually
work items for THIS repo. When that happens, don't just file the entry —
do the work and open a PR.

## When to trigger

The diary entry is about THIS repo if it mentions:
- file paths inside DearDiary (e.g. `install.sh`, `scripts/`, `app/`)
- features that obviously belong here (TODO hooks, diary processor, Tauri
  compose window, the skills system, the install/uninstall flow)
- changes phrased in repo-engineering terms ("update the gitignore",
  "the install script should also …", "the prompt should mention …")

If unsure, file the entry under `diary/tasks/` and DO NOT open a PR. A wrong
PR is worse than no PR — the user can re-trigger by being more explicit.

## What to do

1. **Repo path**: `~/Development/for_fun/DearDiary`. Use `git -C <path>` for
   every git command — the Bash tool's allowlist matches on the leading
   binary, so `cd && git ...` would not be allowed. `gh` reads the repo from
   `cwd` though, so wrap `gh` calls with the `cwd` parameter of the Bash
   tool, OR pass `--repo 0tist/DearDiary` to `gh pr create` explicitly.

2. **Read the entry carefully**. Sometimes the user's instruction is one
   sentence but the implication touches multiple files. Be conservative —
   make the smallest change that captures the request.

3. **Branch from main**:
   ```bash
   git -C ~/Development/for_fun/DearDiary fetch origin main
   git -C ~/Development/for_fun/DearDiary switch -c "diary-fix/<short-kebab-summary>" origin/main
   ```
   Branch prefix `diary-fix/` marks PRs as machine-generated so the user can
   filter them.

4. **Make the change**. Edit files with the Write/Edit tools (these are
   path-agnostic — no `cd` needed). Match existing style: bash scripts use
   `set -u`, tests use the helpers in `tests/lib.sh`, skill files use the
   same frontmatter shape as `learning-finnish/SKILL.md`.

5. **Test verification**: the diary processor doesn't allow running
   `bash tests/run.sh` from inside the cron (Bash allowlist excludes it).
   Don't try. The user's PR review is the verification step. If the change
   really requires test coverage, ADD a new `tests/test_*.sh` so CI / the
   user's local run catches regressions.

6. **Commit** with a clear message:
   ```bash
   git -C ~/Development/for_fun/DearDiary add -A
   git -C ~/Development/for_fun/DearDiary commit -m "$(cat <<'EOF'
   <type>: <one-line subject>

   <body — why, plus a one-line quote of the diary entry that prompted this>

   Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
   EOF
   )"
   ```
   Use the same type prefixes the repo already uses (`feat:`, `fix:`,
   `docs:`, `refactor:`, `test:`).

7. **Push and open a PR**:
   ```bash
   git -C ~/Development/for_fun/DearDiary push -u origin HEAD
   gh pr create --repo 0tist/DearDiary --base main \
     --head "diary-fix/<short-kebab-summary>" \
     --title "<same as commit subject>" \
     --body "$(cat <<'EOF'
   ## Why
   <quote the diary entry verbatim, prefixed by its id>

   ## What changed
   <bullet list of the actual changes>

   ## Test plan
   - [ ] <how to verify>

   🤖 Generated by the deardiary-fixer skill from `~/DearDiary/diary/inbox/<id>.md`
   EOF
   )"
   ```

8. **Switch back to main** so subsequent diary processing doesn't keep
   working on this branch:
   ```bash
   git -C ~/Development/for_fun/DearDiary switch main
   ```

9. **File the diary entry** to `~/DearDiary/diary/tasks/<id>.md` with
   frontmatter additions:
   - `routed_to: tasks`
   - `pr_url: <url returned by gh>`
   - `branch: diary-fix/<…>`
   - `tags: [tasks, deardiary, ...]`

10. **Append a TODO** to `~/DearDiary/TODO.md` Active section:
    `- [deardiary] review PR #<num>: <subject>`

## Hard rules

- NEVER push to `main`. PRs only.
- NEVER force-push, never amend an existing commit on a shared branch.
- NEVER commit secrets, credentials, or local-only paths.
- If the change touches `install.sh` / `uninstall.sh` / `scripts/*.sh`, also
  add or update a `tests/test_*.sh` that covers it.
- If you can't reasonably make the change (need user input, requires manual
  testing, scope creep, ambiguous), file to `diary/tasks/` with a note and
  do NOT open a PR. Append a TODO instead so the user can decide.

## Output

Print exactly one summary line to stdout per processed entry, same as the
default routing — the diary-process.sh script tees this to the events log:

```
<id> -> tasks/<id>.md (PR #<num>)
```

or, if no PR:

```
<id> -> tasks/<id>.md (deferred)
```
